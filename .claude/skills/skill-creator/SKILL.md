# Skill Creator

새 스킬을 작성할 때 따르는 메타 스킬이다.
스킬은 추상 설계가 아니라 실제 반복 작업과 실수 패턴을 압축한 절차여야 한다.

## 절대 규칙

- 즉시 `SKILL.md`를 작성하지 않는다. 먼저 만들고 싶은 작업, 경험 횟수, 실수 패턴을 확인한다.
- 자주 하는 실수가 0개면 스킬 작성 미루기를 권장한다. 추측으로 실패 패턴을 만들지 않는다.
- 추상 원칙을 쓰지 않는다. 증상, 처방, 수치, 파일, grep, 표처럼 검증 가능한 규칙으로 쓴다.

## 작성 전 확인

1. 요청이 "스킬 만들어줘"처럼 비어 있으면 먼저 "어떤 일을 스킬로 만들고 싶나요?"를 묻고 답을 기다린다.
2. `/ask`가 있으면 질문 라운드에 사용한다. 없으면 직접 질문한다.
3. "그 일을 직접 몇 번 해보셨나요?"를 묻는다.
   - 0번: 먼저 2~3번 해보고 실수 패턴을 모은 뒤 다시 만들 것을 권장한다.
   - 1~2번: 실제로 망친 패턴을 구체적으로 묻는다.
   - 여러 번: 좋은 결과물과 실패 패턴을 받아 작성한다.

## 인터뷰 항목

다음 항목을 빠짐없이 확인한다. 모르면 추측하지 말고 다시 묻는다.

| 항목 | 용도 |
|---|---|
| 만들고 싶은 작업/목적 | 범위와 이름 결정 |
| 스킬 이름(kebab-case) | 폴더명과 호출명 |
| 자연어 트리거 3개 이상 | `description` 호출 정확도 |
| 입력 | 파일 경로, 텍스트, 범위 등 |
| 출력 | 생성물과 저장 위치 |
| 잘 나온 결과물 샘플 | 형식과 품질 기준 |
| 자주 하는 실수 3개 이상 | 절대 규칙과 셀프 리뷰 기준 |
| 모드 분기 | 입력별 처리 방식 |

## 작성 형식

```yaml
---
name: <kebab-case-name>
description: >-
  <무엇을 하는 스킬인지>.
  트리거 — '<자연어 1>', '<자연어 2>', '<자연어 3>', '/<slash-command>'.
  입력 — <무엇을 받는지>.
  출력 — <어디에 무엇을 만드는지>.
---
```

본문은 다음 구조를 기본으로 한다.

```markdown
# <스킬 이름>

## 절대 규칙
## 입력 파악
## 구조 규칙
## 영역별 분기
## 콘텐츠 정합성
## 출력 파일
## 셀프 리뷰
```

생성형 스킬이면 디자인 시스템, 템플릿, 샘플 경로를 추가할 수 있다.
큰 자산은 `assets/`, 긴 참고 자료는 `references/`, 좋은 예시 결과물은 `sample/`로 분리하고 본문에는 경로만 둔다.

## 작성 원칙

| 금지 | 사용 |
|---|---|
| 좋은 결과, 적절히, 자연스럽게 | 증상과 처방 |
| 머릿속 확인 | 파일, grep, 표 기반 확인 |
| 일반론 | 실제 Before/After 또는 실패 사례 |
| 과한 설명 | 실행 가능한 단계 |

## 출력 위치

- 프로젝트 전용: `.claude/skills/<skill-name>/SKILL.md`
- 전역: 사용자가 전역/글로벌을 명시한 경우에만 개인 skills 경로
- 경로 명시: 사용자가 준 경로 그대로

## 셀프 리뷰

작성 후 다음을 확인하고 P1/P2로 보고한다.

- P1: `description`에 자연어 트리거 3개 이상과 slash 호출명이 있다.
- P1: 추상어가 남아 있으면 증상-처방 규칙으로 바꿨다.
- P1: 셀프 리뷰가 파일, grep, 표 등 외부화된 검증으로 되어 있다.
- P2: 모드 분기가 입력만 보고 판별된다.
- P2: 출력 형식과 저장 위치가 명확하다.

## 보고 형식

```text
## skill-creator 결과
- 생성 파일:
- 근거 샘플:
- 반영한 실수 패턴:
- 셀프 리뷰:
  - P1:
  - P2:
- 미반영/미검증:
```
