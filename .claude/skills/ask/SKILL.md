# ask

이 skill은 구현/수정/적용을 시작하기 전에 요청의 모호함을 줄이는 데 쓴다.
모호한 요청이면 바로 구현하지 않고 scope, 대상, 검증 기준을 먼저 잠근다.
파일을 만들거나 수정하지 않는다.

## 사용 시점

- 사용자의 목표가 코드 수정, 기능 추가, 평가 중 무엇인지 불명확하다.
- 수정 대상 경로, 보존할 파일, SSOT 문서, 검증 명령이 불명확하다.
- scope, target path, validation 중 하나라도 결과를 크게 바꿀 수 있다.
- 요청 범위가 넓거나 여러 작업이 섞여 있어 바로 수정하면 위험하다.
- 사용자가 "알아서", "적당히", "이거 처리", "좋게 바꿔"처럼 판단 기준을 충분히 주지 않았다.

## 사용하지 않는 경우

- 사용자가 수정 대상, 범위, 검증 기준을 명확히 줬다.
- 사용자가 합리적 기본값으로 바로 진행하라고 명시했다.
- 단순 조회, 단순 설명, 단일 파일의 명확한 수정처럼 추가 질문이 필요 없다.

## 진행 방식

1. 현재 이해한 요청을 한 문장으로 요약한다.
2. 이미 있는 문서나 파일에서 바로 확인할 수 있는 정보는 먼저 확인한다.
3. 그래도 결정이 필요한 항목만 1~3개 질문한다.
4. 가능한 경우 권장 선택지를 먼저 제시하고 이유를 짧게 덧붙인다.
5. 질문은 scope, 대상 경로, 검증 기준, 보존/수정 파일 중심으로 제한한다.
6. 답을 받기 전에는 파일을 만들거나 수정하지 않는다.

## 출력 형식

```text
## 확인 필요

제가 이해한 요청:
- <한 문장 요약>

기본 판단:
- <권장 방향 또는 확인된 사실>

질문:
1. <꼭 필요한 질문>
2. <선택 질문>
3. <선택 질문>
```
