# Phase 1: 인증 + 가족 그룹/멤버십 Design Review

> 상태: 진행 중 · 범위: Phase 1 뼈대(모든 권한이 여기 얹힘) · SSOT: `CLAUDE.md` 3·5·6·10장, `lib/models/`

## 목표
- 사용자가 이메일로 로그인/회원가입하고, 로그인 후 "어느 아이로 들어갈지" 선택할 수 있다.
- 부모가 **새 아이를 등록**하면 아이([Child]) + 그룹([FamilyGroup]) + 본인 멤버십(PARENT·ACTIVE·isAdmin)이 한 번에 생성된다.
- 다른 사용자가 **초대 코드로 참여**하면 PENDING 멤버십이 생기고, 관리자가 승인해야 ACTIVE가 된다.
- 모든 그룹 데이터 접근은 멤버십을 거쳐 검증된다(역할·상태 게이트). 이후 Phase 2+ 기능이 이 위에 얹힌다.

## 사용자 시나리오
- 주 시나리오(부모, 최초): 회원가입 → 시작 선택 화면 → "새 아이 등록" → 아이 정보 입력 → 그룹 자동 생성 → 내 역할 PARENT(관리자)로 진입.
- 주 시나리오(배우자/친척, 합류): 로그인 → "초대 코드로 참여" → 코드 입력 → PENDING 상태로 대기 → 관리자 승인 후 시작 선택 화면에 해당 아이가 뱃지(부모/친척)와 함께 노출.
- 주 시나리오(관리자, 승인): 가족 그룹·초대 화면에서 PENDING 멤버 확인 → 승인(ACTIVE) 또는 거절 → 필요 시 초대 코드 재발급.
- 보조 시나리오: 한 계정이 그룹 A에선 PARENT, 그룹 B에선 RELATIVE. 시작 선택 화면은 내가 ACTIVE인 모든 그룹을 역할 뱃지와 함께 나열한다.

## 입력 / 출력
- 입력: 이메일/비밀번호, 아이 정보(name·birthDate·sex·선택 필드), 초대 코드, 승인/거절 액션.
- 출력/상태 변경:
  - `users/{uid}` upsert(최초 로그인 시 [AppUser] 생성).
  - 아이 등록 → `children/{childId}` + `groups/{groupId}`(inviteCode 포함) + `memberships/{groupId}_{uid}`(PARENT·ACTIVE·isAdmin=true) **배치 생성**.
  - 코드 참여 → `memberships/{groupId}_{uid}`(RELATIVE·PENDING) 생성.
  - 승인 → 해당 멤버십 status PENDING→ACTIVE.
- 부수 효과: Firebase Auth 세션. 잘못된 접근은 Firestore 규칙에서 거부(서버 강제).

## 제약
- 기술: Flutter + Firebase(auth/firestore). 멤버십 문서 ID = `{groupId}_{userId}` → 규칙에서 조인 없이 `get`/`exists` 한 번으로 검증. 기록/게시물은 `groupId` 비정규화(Phase 2+).
- 보안: 공개범위·역할은 UI가 아니라 **Firestore 규칙에서 강제**. PENDING은 그룹 데이터 접근 불가. 권한 폴백은 최소 권한(role→relative, status→pending).
- 비즈니스: 가족 실사용 우선(상용 출시 아님). 아이 1명당 그룹 1개. 보안 규칙은 Phase 1부터 같이 깐다.

## 엣지케이스
- 이미 ACTIVE/PENDING 멤버인 그룹에 같은 코드 재입력 → 중복 멤버십 생성 금지(문서 ID 합성으로 자연 차단, 기존 status 보존).
- 존재하지 않거나 폐기된(재발급된) 초대 코드 입력 → 명확한 실패 메시지.
- 아이 등록 도중 부분 실패 → child/group/membership는 **WriteBatch로 원자적 생성**(부분 생성 방지).
- 관리자가 0명이 되는 상황(유일 관리자 탈퇴/강등) → Phase 1에서는 마지막 isAdmin 제거를 막는다(또는 탈퇴 보류). 상세는 미해결 질문.
- 동시 승인/중복 승인 → 멱등 처리(이미 ACTIVE면 no-op).
- 자기 자신을 강등/제거 시도, 비관리자가 멤버 관리 시도 → 규칙·UI 양쪽 차단.

## 검증 기준
- 기능:
  - 신규 가입 → 아이 등록 1회로 child/group/membership 3개 문서가 일관되게 생성된다(배치).
  - 코드 참여 후 status=PENDING이며, 승인 전에는 그룹 문서 읽기가 규칙에서 거부된다.
  - 승인 후 그룹 데이터 읽기가 성공한다.
  - RELATIVE는 아이 정보/기록 쓰기가 규칙에서 거부된다.
- 비기능:
  - 모델 직렬화 라운드트립 유닛 테스트 통과(`flutter test`).
  - `flutter analyze` 무결.
  - 보안 규칙은 에뮬레이터/규칙 테스트로 멤버십 게이트 검증(가능 범위).
  - rrn 등 민감정보는 일반 조회 경로에 평문/마스킹을 싣지 않는다.

## Out of Scope (Phase 1)
- 소셜 로그인(구글/애플), 비밀번호 재설정 UX 고도화.
- 주민등록번호 암호화/복호화 함수 및 재인증(Phase 3, 아이 정보 카드).
- 성장/수면/접종/피드/채팅 등 기능 화면(Phase 2+).
- 초대 코드 만료·푸시 알림.

## 미해결 질문
- 인증 수단: Phase 1은 이메일/비밀번호만으로 간다(가족 실사용). 소셜 로그인은 후순위 — 이대로 진행. → **결정: 이메일/비밀번호**.
- 초대 코드 형식/충돌 처리: 짧은 대문자+숫자 코드 생성, 충돌 시 재생성. 길이/문자셋은 구현 시 6~8자 영숫자로 잠정 결정.
- 유일 관리자 보호 정책의 정확한 UX(탈퇴 차단 vs 위임 강제)는 멤버 관리 화면 구현 시 확정.
